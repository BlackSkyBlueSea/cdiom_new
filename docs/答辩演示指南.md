# CDIOM 毕业设计答辩演示指南

本文档汇总环境准备、数据库脚本路径、服务启动、多角色演示流程及应急预案，与第六章「系统测试」用例（CORE-INV、CORE-PO、CORE-IN、CORE-OUT、INT-FE、NFR 等）对应，便于答辩现场按步骤复现。

## 1. 演示目标与范围

| 目标 | 说明 |
|------|------|
| 环境可复现 | 固定 JDK 17、Node、MySQL、前后端端口与配置 |

## 2. 环境准备

### 2.1 推荐软件版本

| 组件 | 版本/说明 |
|------|-----------|
| 操作系统 | Windows 10+ |
| Java | **17**（与 `pom.xml` 中 `java.version` 一致） |
| Maven | 3.9.x（示例路径 `D:\apache-maven-3.9.5`） |
| Node.js | 建议 **22.x**（与项目 `package-lock.json` 一致；可用 nvm 管理） |
| MySQL | **8.0.x**，库名 **`cdiom_db`** |
| 浏览器 | Chrome / Edge（答辩前关闭无关扩展，减少控制台噪音） |

### 2.2 环境变量（本机 D 盘示例）

答辩前在 **新开的 CMD 或 PowerShell** 中确认（或写入用户环境变量后重开终端）：

| 变量名 | 示例值 |
|--------|--------|
| `JAVA_HOME` | `D:\jdk-17.0.5`（**不要**写成 `...\bin`） |
| `MAVEN_HOME` | `D:\apache-maven-3.9.5` |
| `NVM_HOME` | `D:\Roaming\nvm` |
| `NVM_SYMLINK` | `D:\nvm4w\nodejs`（需先 `mkdir D:\nvm4w`，再 `nvm use`） |

**Path** 需包含：`%JAVA_HOME%\bin`、`%MAVEN_HOME%\bin`、`%NVM_HOME%`、`%NVM_SYMLINK%`。

验收命令：

```bat
java -version
mvn -v
node -v
npm -v
```

### 2.3 PowerShell 与 npm（常见拦截）

若执行 `npm run dev` 报「禁止运行脚本」，任选其一：

- 使用：`npm.cmd run dev`
- 或：`Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned`

### 2.4 后端配置

- 主配置：`cdiom_backend/src/main/resources/application.yml`（端口 `8080`、库 `cdiom_db`）
- 本地敏感项：`cdiom_backend/src/main/resources/application-local.yml`（数据库账号密码、JWT、邮箱、第三方 API Key）

答辩前确认：`spring.datasource.username/password` 与现场 MySQL 一致。

### 2.5 前端配置

- 开发代理：`cdiom_frontend/vite.config.js` 中 `/api` → `http://localhost:8080`
- 前端开发地址：**`http://localhost:5173`**

## 3. 数据库脚本路径与执行顺序

### 3.1 全新库（首次或彻底重建）

在 **`cdiom_backend`** 目录下执行（MySQL 客户端路径按本机安装调整）：

```bat
cd /d F:\Three\GT\ProceduralFiles\cdiom_new\cdiom_backend

"C:\Program Files\MySQL\MySQL Server 8.0\bin\mysql.exe" -uroot -p你的密码 --default-character-set=utf8mb4 --show-warnings -e "source src/main/resources/db/init_simple.sql"

"C:\Program Files\MySQL\MySQL Server 8.0\bin\mysql.exe" -uroot -p你的密码 cdiom_db --default-character-set=utf8mb4 -e "source src/main/resources/db/init_permissions.sql"
```

在 **PowerShell** 中请将上述两条里的可执行文件调用改为：`& "C:\Program Files\...\mysql.exe" ...`（见 3.2 节 PowerShell 示例）。

> 说明：`init_simple.sql` 内含 `CREATE DATABASE` 与 `USE cdiom_db`，首条命令**不要**在 `-e` 里预先指定 `cdiom_db`，否则会报「Unknown database」。  
> 其余迁移脚本若此前已全量执行过，可跳过；若从零搭建，请参考 `src/main/resources/db/README.md` 中的顺序。

### 3.2 答辩专用演示数据（推荐每次彩排前执行）

脚本目录（相对仓库根）：

```
cdiom_backend/src/main/resources/db/defense/
├── 00_defense_cleanup.sql   # 清理 DF26 前缀演示数据
├── 01_defense_seed.sql      # 写入演示用户、供应商、药品、库存、订单、出库申请
└── 02_defense_verify.sql    # 验收查询
```

说明：

- `00_defense_cleanup.sql` 已内置 `SQL_SAFE_UPDATES` 兼容处理，可在 MySQL Workbench 安全模式下直接执行（避免 Error 1175）。
- `01_defense_seed.sql` 采用幂等写法（`ON DUPLICATE KEY UPDATE` + 明细先删后插），可重复执行用于多次彩排重置。

在 **`cdiom_backend`** 下执行（**必须按顺序**）。

**CMD（可直接复制）**：

```bat

"C:\Program Files\MySQL\MySQL Server 8.0\bin\mysql.exe" -uroot -p你的密码 cdiom_db --default-character-set=utf8mb4 -e "source src/main/resources/db/defense/00_defense_cleanup.sql"

"C:\Program Files\MySQL\MySQL Server 8.0\bin\mysql.exe" -uroot -p你的密码 cdiom_db --default-character-set=utf8mb4 -e "source src/main/resources/db/defense/01_defense_seed.sql"

"C:\Program Files\MySQL\MySQL Server 8.0\bin\mysql.exe" -uroot -p你的密码 cdiom_db --default-character-set=utf8mb4 -e "source src/main/resources/db/defense/02_defense_verify.sql"
```

**PowerShell（路径含空格时必须加调用运算符 `&`，且 `-p` 与密码之间不要空格）**：

## 5. 多角色演示：推荐窗口策略

### 5.1 推荐方式（与当前前端实现一致）

- **窗口 A（主讲）**：登录 `super_admin` 或 `admin`，地址 `http://localhost:5173/app`
- **窗口 B（协同）**：在 A 中点击右上角 **「登录其他用户」**，新标签打开登录页，登录 `buyer_df` / `wh_admin_df` / `nurse_df` 等

**不要**同时开 4～5 个随意新窗口各登一遍，易与 Cookie/session 混淆；**两个窗口**足够覆盖多角色流程。

### 5.2 会话自检（可选，任一窗口 F12 控制台）

```js
fetch('/api/v1/auth/current', {
  headers: { Authorization: 'Bearer ' + sessionStorage.getItem('cdiom_token') }
}).then(r => r.json()).then(console.log)
```

A、B 各执行一次，应分别看到对应 `username`、`roleId`。

### 5.3 已修复的多用户问题（答辩前请拉最新前端）

仓库内已调整：

- `cdiom_frontend/src/utils/auth.js`：普通登录清除 `cdiom_multi_login` 标记；`removeToken` 在多用户模式下避免误清全局 Cookie  
- `cdiom_frontend/src/utils/request.js`：多用户标签下 401 不清 Cookie，避免误踢另一窗口

答辩前请 **重新构建/刷新前端** 后再彩排「A 登录 → B 登录其他用户 → A 退出再登录」全流程。

## 6. 详细演示流程（建议总时长 12～15 分钟）

下列步骤与论文章节对应关系在括号中标注。

### 阶段 0：开场（30 秒）

说明：本系统为临床药品出入库管理，演示覆盖库存与预警、采购与条码、入库、出库全链路、参数联动及安全边界；数据为答辩专用 `DF26` 前缀，可一键清理与重建。

### 阶段 1：管理员 / 超级管理员 — 库存与预警（A 窗口）

| 顺序 | 操作 | 对应用例 / 说明 |
|------|------|-----------------|
| 1.1 | 登录 `super_admin` / `super123`（或 `admin` / `admin123`） | — |
| 1.2 | 进入 **库存管理**，按药品名关键字「答辩」、批次 `DF26-BATCH`、效期区间筛选 | CORE-INV-01 |
| 1.3 | 打开 **近效期预警**（或接口页面/Network 展示 `near-expiry` 类请求） | CORE-INV-02 |

### 阶段 2：采购专员 — 订单与条码（B 窗口）

| 顺序 | 操作 | 对应用例 |
|------|------|----------|
| 2.1 | B 登录 `buyer_df` / `admin123` | — |
| 2.2 | 进入 **采购订单**，可基于已有 `DF26-PO-001/002` 讲解状态；如需现场新建订单再演示创建 | CORE-PO-01 |
| 2.3 | 对订单执行 **条码生成/下载** | CORE-PO-02 |

### 阶段 3：仓库管理员 — 入库与库存变化（B 窗口切换账号）

| 顺序 | 操作 | 对应用例 |
|------|------|----------|
| 3.1 | B 退出，登录 `wh_admin_df` / `admin123` | — |
| 3.2 | **从采购订单入库**（选择 `DF26-PO-001` 及明细，填写批次、效期、数量等合法数据） | CORE-IN-01 |
| 3.3 | 演示 **效期校验**（界面或接口返回 PASS/WARNING/FORCE 等分级） | CORE-IN-02 |
| 3.4 | **临时入库**（无订单路径，与订单入库对比） | CORE-IN-03 |
| 3.5 | 回到 **库存**，对比入库前后数量 | INT-FE-01 |

### 阶段 4：医护人员 — 出库申请（B 窗口）

| 顺序 | 操作 | 对应用例 |
|------|------|----------|
| 4.1 | B 退出，登录 `nurse_df` / `admin123` | — |
| 4.2 | 进入 **出库管理**，说明已有 `DF26-OUT-001`（正常）、`DF26-OUT-002`（超库存）；可再新建一条申请演示表单 | CORE-OUT-01 |

### 阶段 5：仓库管理员 — 审批、预检、执行（B 窗口）

| 顺序 | 操作 | 对应用例 |
|------|------|----------|
| 5.1 | B 登录 `wh_admin_df` | — |
| 5.2 | 对 `DF26-OUT-001` 做 **库存预检**（页面或 Network 展示 `stock-check`） | CORE-OUT-03 |
| 5.3 | **审批通过** 再 **执行出库**（说明审批阶段不扣库存，执行后扣减；可多批次体现 FIFO） | CORE-OUT-04 |
| 5.4 | 对 `DF26-OUT-002` 审批或执行，展示 **库存不足** 友好提示且 **不部分扣减** | CORE-OUT-02、CORE-OUT-05 |
| 5.5 | 库存页对比执行前后数量 | 与图 6-12～6-24 类似 |

### 阶段 6：参数联动（A 窗口）

| 顺序 | 操作 | 对应用例 |
|------|------|----------|
| 6.1 | A 进入 **参数配置**，修改 `expiry_warning_days`、`expiry_critical_days` | INT-FE-03 |
| 6.2 | 刷新 **仓库工作台 / 预警** 展示变化 | 与论文图 6-25～6-27 对应 |

### 阶段 7：安全与非功能（A 或 B）

| 顺序 | 操作 | 对应用例 |
|------|------|----------|
| 7.1 | 未带 Token 调用受保护接口（Postman 或浏览器无登录） | NFR-SEC-01 → 401 |
| 7.2 | `nurse_df` 访问 **用户管理** 等无权限菜单/接口 | NFR-SEC-02 → 403 或友好提示 |
| 7.3 | （可选）上传超大文件或非允许类型 | NFR-SEC-03 |
| 7.4 | （可选）库存列表分页响应时间口述 | NFR-PERF-01 |

### 阶段 8：收尾（30 秒）

总结：核心业务闭环、规则与数据一致、前后端契约清晰；局限为自动化单测与压测基线可后续建设。

## 8. 应急预案

| 现象 | 处理 |
|------|------|
| `401 未登录或登录已过期` | 当前标签 `sessionStorage` 无 token；重新登录；两窗口分别自检 `/auth/current` |
| `403 权限不足` | 确认当前账号角色；超级管理员用 `super_admin` |
| 第三方药品 API 失败 | 改演示「本地药品检索」路径，不依赖外网 |
| 数据被改乱 | 重新执行 `00_defense_cleanup.sql` + `01_defense_seed.sql` |
| 控制台 `runtime.lastError` | 多为浏览器扩展问题，可忽略或无痕模式重试 |
